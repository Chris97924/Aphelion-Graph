## Context

Aphelion v0.4 ships a stable wire format (`format_version: "2.0"`) with required claim frontmatter (`claim_id`, `state`, `created_at`, etc.) and a small set of optional fields (`confidence`, `subject`, `predicate`, `object`, `tags`). The Aphelion canonical roadmap (Notion `347f3661...`) reserves R1-R4 slots for claim-level semantic enrichment to support consumer-side conflict resolution — required by the Apex M5 dual-read adapter (Parallax-Kernel side).

The roadmap memo proposed five additive fields and a reader-side `conflict_class` derivation. A first xcouncil round (2026-05-09 morning) drafted the schema. This ADR records the second-round resolution of six §9 unresolved items, with verdicts derived from an 8-model independent consensus.

## Rationale by item

### 1. confidence precision: 3dp (spec-evidence override)

The v0.3 memo proposed 2dp on grounds of "minimal precision sufficient". Reading `spec/claim-frontmatter.md` line 20 showed v0.4 already mandates "exactly 3 decimal digits (e.g. 0.900, not 0.9)". Adopting 2dp would break round-trip with all v0.4 producers. The spec is the source of truth — not the memo. Validator emits `PX_E_CLAIM_CONFIDENCE_PRECISION` for non-3dp wire format.

### 2. supersedes cross-package: lenient

Aphelion is a multi-package corpus format. Strict cross-package validation would force readers to load every referenced package upfront — impractical for partial loads, demand-loaded vaults, or detached snapshots. The legitimate use case "package A contains a claim that supersedes package B's older claim" must be supported. Lenient + explicit warning preserves data integrity signals without aborting reads.

### 3. key order: raise + CLI

Auto-normalize-on-read (option b) would silently mask writer bugs and create disagreement between on-disk bytes and content_hash inputs. Strict raise + a one-shot `aphe canonicalize` CLI keeps the contract loud while making the fix cheap. This matches the JSON-canonicalization tradition (Aphelion already enforces canonical JSON for content hashes).

### 4. query_time default: now(UTC) + echo

Required-pass (option a) burdens every caller with a `datetime.now()` call. None=all-time (option c) is a footgun — a query meant for "current state" silently returns expired claims. Default `now(UTC)` matches the dominant intent ("what's true right now?"). Echoing `used_query_time` in the payload restores reproducibility — any caller can log the value and re-issue the same query later for byte-identical R4 output.

### 5. R4 needs subject (primary design call — drove the spec)

R4's semantic basis is "claims about the same subject can conflict". Without a subject, there is no grouping key, and `conflict_class` derivation has no meaning. The four rejected options have specific failure modes:

- **(a) skip on missing subject** — claims that explicitly opt into R1-R4 fields would be silently inert; surprising for writers.
- **(b) composite fallback** — `predicate` and `object` are also `format-optional`, cascading to undefined behavior.
- **(d) synthetic = claim_id** — every claim becomes its own group, R4 always returns `none`, killing the feature.

Conditional requirement preserves backward compatibility for the v0.3-net-new trigger fields (`polarity`, `valid_from`, `valid_until`, `supersedes`): v0.4 claims without those fields are unaffected. `subject` itself remains `format-optional` at the schema-matrix level — the conditional check is a v0.3 validator concern, not a wire-format change.

### Backward-compat carve-out — `confidence` excluded (Chris-confirmed 2026-05-09 PM)

The original D1.5 verdict text listed five R4-trigger fields: `confidence | polarity | valid_from | valid_until | supersedes`. Phase-4 review identified that `confidence` is a long-standing v0.4 `format-optional` field (`claim-frontmatter.md` line 20, normative since 2026-04-21). Including it as a trigger would have made every existing v0.4 claim using `confidence` without `subject` a v0.3 validation error — a hard breaking change, contradicting the "additive only" framing in §1 of `spec/v0.3-claim-semantics.md`.

**Chris confirmed 2026-05-09 PM that `confidence` stays excluded.** Final R4-trigger list is the four v0.3-net-new fields (`polarity`, `valid_from`, `valid_until`, `supersedes`).

Rationale (recorded for future reference): `confidence` is metadata about a single claim's certainty — it is a tiebreak input to R4 conflict resolution, not an opt-in signal that a claim participates in a conflict graph. The other four fields actively declare conflict-graph participation; `confidence` does not. Writers who want subject-required semantics around a confidence-bearing claim must opt in explicitly by also setting one of the four trigger fields.

### 6. polarity-divergence supersession: deferred

Conflating "I disagree with you" (polarity divergence) with "I replace you" (supersession) would lose information. R4 surfaces `contradiction` for the former and `supersession` for the latter — that distinction is load-bearing. The Nemotron/Laguna proposal is a real future case (when polarity differs AND temporal containment holds, the newer might semantically replace the older), but its boundary cases are intricate (partial containment? mixed polarity within a chain?) and adding it now blows v0.3's YAGNI envelope. Track as v0.4 candidate.

## Consequences

### Positive

- **Reader-side derivation** — no wire-format bump, no v0.4 producer breakage
- **Conditional requirement** — minimal v0.4 friction; subject stays format-optional
- **Canonical CLI** — `aphe canonicalize` gives writers a single official tool
- **Explicit determinism path** — `used_query_time` echo lets callers replay R4 byte-identically
- **Lenient cross-package** — multi-package corpora work without forced eager loading

### Negative

- **Two precision sources of truth on `confidence`** — v0.4 schema-matrix says `format-optional` only; v0.3 spec adds 3dp + range + type checks. Mitigated by adding error codes to `error-codes.md` and cross-referencing from both spec docs.
- **Conditional requirement** is a v0.3 validator behavior, not visible to v0.4 readers. A v0.4 reader processing a v0.3-violating claim (R4 trigger present, no subject) will not catch it. Mitigated by the validator being part of `aphe verify` and by `aphe canonicalize` rejecting such inputs with `PX_E_CLAIM_SUBJECT_REQUIRED_FOR_CONFLICT`.

### Neutral

- 14 new error codes (`PX_E_4101`..`PX_E_4144`) + 1 warning (`PX_W_4151`) — registry growth proportional to feature surface.
- ADR establishes `PX_W_*` warning prefix convention for the registry.
