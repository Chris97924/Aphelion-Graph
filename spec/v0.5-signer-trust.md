# Aphelion Signer / Trust (v0.5)

**Version:** 0.5.0-draft
**Status:** Draft
**Date:** 2026-04-27
**Reserved-by:** R7 (Roadmap §RESERVED)

This document specifies the v0.5 signer / trust extension. It is **additive**
to v0.4 wire format: a v0.4 reader that ignores `signatures.jsonl` and the
`signers/` directory MUST still validate every previously-conforming v0.4
package unchanged.

## 1. Roles

Aphelion separates three roles. One identity may fill more than one role on
the same package, but the role distinction MUST be preserved in metadata.

| Role | Question it answers | Required? |
|---|---|---|
| `author` | Who composed the claim body? | Already mandated by v0.3 frontmatter. |
| `approver` | Who reviewed the claim and accepted it for distribution? | Optional; per-claim. |
| `signer` | Who attests, with a cryptographic signature, that the package bytes are unaltered? | Optional; per-package. |

A single signer signs the **package** as a whole; per-claim signatures are
out of scope for v0.5.

### 1.1 `author`

Carried in `claim.md` frontmatter `author` (v0.3, normative). v0.5 does not
change this field.

### 1.2 `approver`

New optional frontmatter field `approvers: [string, ...]` on `claim.md`.
Each entry MUST be a non-empty UTF-8 string. Empty list and missing field
are equivalent. Approver identity strings are opaque to the validator.

### 1.3 `signer`

A signer is identified by a `signer_id`, an opaque non-empty UTF-8 string,
plus a public key fingerprint (algorithm-specific, see §3). Signer
metadata lives in `signers/<signer_id>.json` inside the tar archive
(see §2.3).

## 2. Signature Envelope

### 2.1 Bound bytes

A signature is computed over the **package canonical hash**:

```
package_canonical_hash = sha256(
    json_canonical([
        format_version,
        package_id,
        sorted([
            (claim_id, claim_instance_id, content_hash)
            for every claim in package
        ]),
    ])
).hexdigest()
```

This binds the signature to (a) the wire format major.minor, (b) the
package identity, and (c) the exact set of claim instances and their
content hashes. Re-signing without re-packaging is forbidden — any change
that affects the canonical hash invalidates the signature.

`json_canonical` follows v0.2 §S2.1 (UTF-8, no BOM, lex key sort,
`(",", ":")` separators, no `NaN`/`Infinity`).

### 2.2 Envelope fields

A `SignatureEnvelope` is a frozen record with exactly these fields:

| Field | Type | Description |
|---|---|---|
| `signer_id` | string | §1.3 |
| `algorithm` | string | §3.1 registry name |
| `signed_at_iso` | string | RFC 3339 UTC, **millisecond** precision (3 sub-second digits) per `spec/canonical-serialization.md` Rule 4, trailing `Z` (e.g. `2026-04-27T00:00:00.000Z`) |
| `package_canonical_hash` | string | §2.1, hex lowercase, length 64 |
| `signature_b64` | string | algorithm-specific signature bytes, base64-standard, no padding stripped |

The envelope is **append-only** in `signatures.jsonl` at the package root.
Multiple envelopes MAY exist for the same package (multi-signer); each line
is one envelope, written canonically by §2.4.

### 2.3 `signers/<signer_id>.json`

Each referenced `signer_id` MUST have a manifest at
`signers/<signer_id>.json`:

| Field | Type | Description |
|---|---|---|
| `signer_id` | string | MUST equal the path stem |
| `algorithm` | string | §3.1 registry name |
| `public_key_b64` | string | base64-standard public key bytes |
| `key_fingerprint` | string | `sha256(<public_key_bytes>)` hex lowercase, length 64 |
| `notary_uri` | string \| null | optional §4 reference |

The `key_fingerprint` MUST be recomputed by the verifier; mismatched
fingerprint → `E_SIGNER_FINGERPRINT_MISMATCH`.

### 2.4 Canonical line layout

Each `signatures.jsonl` line is one `SignatureEnvelope` serialized as
canonical JSON (v0.2 §S2.1) followed by `\n`. The file MUST end with
exactly one `\n`. Lines MUST be sorted lex-ascending by `signer_id` then
`signed_at_iso`. Rationale: deterministic round-trip — repacking the same
package with the same envelopes produces byte-identical
`signatures.jsonl`.

## 3. Algorithms

### 3.1 Algorithm registry

| Name | Public-key algorithm | Notes |
|---|---|---|
| `ed25519` | RFC 8032 Ed25519 | DEFAULT for production |
| `hmac-sha256` | RFC 2104 with SHA-256 | TEST-ONLY; NOT a public-key algorithm; provided for fixture authoring and reference-implementation testing |

The validator MUST reject any envelope whose `algorithm` is not in the
registry → `E_SIGNER_ALGORITHM_UNKNOWN`.

### 3.2 `hmac-sha256` semantics

For `hmac-sha256`, `public_key_b64` carries the shared secret bytes.
This algorithm has **zero non-repudiation properties** and MUST NOT be
used for production attestation. Validators MUST emit a warning when
verifying an `hmac-sha256` envelope unless the caller explicitly enables
`--allow-test-algorithms`.

Why include it: v0.5 reference implementation and fixture suite need a
deterministic, dependency-free algorithm to exercise envelope round-trip,
canonical-line ordering, and tamper detection without taking on a crypto
library dependency in the core package.

### 3.3 `ed25519` semantics

Standard Ed25519 over the raw bytes of `package_canonical_hash` decoded as
hex. Public key MUST be the 32-byte raw form, base64-standard encoded.
Signature MUST be the 64-byte raw form, base64-standard encoded.

A real Ed25519 implementation requires a non-stdlib dependency
(`cryptography` ≥ 42 or `pynacl` ≥ 1.5). v0.5 ships the algorithm-agnostic
core under `aphelion[signer]` extra; consumers without the extra get the
`hmac-sha256` reference path only.

## 4. Trust Notary (Extension Point)

A trust notary is an external service that vouches for a signer's identity
binding (e.g., that `signer_id="alice"` in this package is the same Alice
who controls public key `<fingerprint>`). v0.5 defines only the
**extension point**, not a normative resolver protocol.

If a `signers/<signer_id>.json` declares `notary_uri`, validators MAY
query it to obtain a notary attestation (out-of-band; URI scheme,
authentication, and response shape are NOT specified in v0.5).
Validators that do not understand or cannot reach `notary_uri` MUST treat
the envelope as `verified-locally` only.

Reserved for v0.6+: notary attestation envelope format, revocation list
distribution, multi-notary quorum.

## 5. Verification Rules

A package is **signed-valid** iff:

1. The package validates under v0.4 rules (§S1–§S3 in roadmap; v0.4
   wire). Failure here is reported with v0.4 error codes; signatures are
   not even checked.
2. For every line in `signatures.jsonl`:
   - The envelope parses against §2.2 schema → else
     `E_SIGNATURE_MALFORMED`.
   - The referenced `signers/<signer_id>.json` exists and parses against
     §2.3 → else `E_SIGNER_MISSING` or `E_SIGNER_MALFORMED`.
   - `key_fingerprint` recomputes to the declared value → else
     `E_SIGNER_FINGERPRINT_MISMATCH`.
   - The envelope's `package_canonical_hash` recomputes against the
     current package contents → else `E_SIGNATURE_HASH_MISMATCH`.
   - `algorithm` is in the §3.1 registry → else
     `E_SIGNER_ALGORITHM_UNKNOWN`.
   - The signature bytes verify against the public key over the canonical
     hash → else `E_SIGNATURE_INVALID`.
3. Lines in `signatures.jsonl` are sorted per §2.4 → else
   `E_SIGNATURE_ORDER`.

A package with NO `signatures.jsonl` and NO `signers/` directory remains
**unsigned-valid** under v0.4 — v0.5 introduces no new MUST-have file.

## 6. Error Codes (additive)

| Code | Meaning |
|---|---|
| `E_SIGNATURE_MALFORMED` | Envelope schema violation. |
| `E_SIGNATURE_HASH_MISMATCH` | Envelope hash ≠ recomputed canonical hash. |
| `E_SIGNATURE_INVALID` | Cryptographic verification failed. |
| `E_SIGNATURE_ORDER` | `signatures.jsonl` lines not sorted per §2.4. |
| `E_SIGNER_MISSING` | Envelope references a `signer_id` with no manifest. |
| `E_SIGNER_MALFORMED` | `signers/<signer_id>.json` schema violation. |
| `E_SIGNER_FINGERPRINT_MISMATCH` | Declared fingerprint ≠ recomputed. |
| `E_SIGNER_ALGORITHM_UNKNOWN` | `algorithm` not in §3.1 registry. |

## 8. Migration

v0.4 → v0.5 is **strictly additive**. No `aphe migrate` step is required;
v0.4 packages remain valid v0.5 packages. A package author who wants to
sign an existing v0.4 package can:

1. Copy or re-pack the v0.4 package (preserves canonical hash).
2. Add `signers/<id>.json` and `signatures.jsonl` per §2.
3. Re-pack to embed the new files.

The `package_id` MUST NOT change across this transition; the
`package_canonical_hash` per §2.1 changes only because the claim set is
unchanged but the file layout grows new entries. (Note: the canonical
hash binds claim content, NOT archive layout — see §2.1 input list.)

## 10. Open Questions (Phase A defers)

- Q1: Does `package_canonical_hash` need to commit to the **algorithm**
  used (i.e., domain separation against future hash bumps)? Current draft
  says no because the envelope itself names the algorithm; revisit if any
  multi-algorithm cross-attack is identified.
- Q2: Should `signed_at_iso` be inside the signed bytes? Current draft
  says no — the bound bytes are exactly `package_canonical_hash`, and
  `signed_at_iso` is metadata that cannot be retroactively forged because
  it lives inside the envelope that itself is referenced by the
  signature. Revisit if a replay-style attack vector emerges.
- Q3: Should `hmac-sha256` envelopes be allowed to round-trip through
  `signatures.jsonl` at all in production validators, or only in test
  mode? Current draft permits storage but requires a warning at
  verification time. Revisit after Lane C v0.5 redaction guideline lands.
