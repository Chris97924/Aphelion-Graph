# Aphelion Notary Attestation (v0.6, R7.2)

**Version:** 0.6.0
**Status:** Final
**Date:** 2026-06-20
**Reserves:** R7.2 (`spec/v0.5-signer-trust.md` ┬¦7) ŌĆö *Notary attestation envelope format*

This document specifies the **notary attestation envelope format** and its
**verification path**, the item reserved as R7.2 for v0.6 in the v0.5 signer /
trust spec. It is strictly **additive**: a package with no notary attestation
behaves exactly as in v0.5 (`resolve_notary` yields `"verified-locally"`), and
no new file is mandatory.

R7.1 (per-claim signatures) and R7.3 (revocation list distribution) remain
reserved and out of scope for this document.

## 1. Background (v0.5 extension point)

`spec/v0.5-signer-trust.md` ┬¦4 defines the trust notary as an *extension
point* only: a `signers/<signer_id>.json` MAY carry `notary_uri`, and a
validator that cannot understand or reach the notary MUST treat the signature
as `verified-locally`. v0.5 specified neither the wire shape of a notary's
response nor how a validator turns that response into a
`"verified-by-notary"` attestation. This document fills exactly that gap.

A trust notary vouches for a **signer identity binding**: that the public key
(by fingerprint) carried in `signers/<signer_id>.json` is the key the notary
has independently associated with that `signer_id`. A notary does **not**
re-sign package content; the signer's own `signatures.jsonl` envelope already
binds the package canonical hash (┬¦2.1 of v0.5).

## 2. Attestation Envelope

### 2.1 Bound bytes

A notary attestation is a detached signature computed over the
**attestation canonical hash**:

```
attestation_canonical_hash = sha256(
    json_canonical({
        "key_fingerprint": <signer key_fingerprint, ┬¦2.3 v0.5>,
        "notary_id":       <opaque non-empty UTF-8 notary identity>,
        "signer_id":       <signer_id being vouched for>,
    })
).hexdigest()
```

`json_canonical` follows `spec/canonical-serialization.md` (UTF-8, no BOM,
lex key sort, `(",", ":")` separators, no `NaN`/`Infinity`, trailing `\n`).

This binds the attestation to (a) the exact signer identity string, (b) the
exact public-key fingerprint, and (c) the notary's own identity. It does
**not** bind any package hash: a single notary attestation for a key is
reusable across every package that key signs, which is the intended
deployment shape (the notary vouches for the *keyŌåÆidentity* binding, signed
content is the signer's job).

### 2.2 Envelope fields

A `NotaryAttestationEnvelope` is a frozen record with exactly these fields:

| Field | Type | Description |
|---|---|---|
| `notary_id` | string | Opaque non-empty UTF-8 notary identity. |
| `signer_id` | string | The `signer_id` whose key the notary vouches for; MUST equal the signer manifest `signer_id`. |
| `key_fingerprint` | string | The signer key fingerprint the notary vouches for; `sha256` hex lowercase, length 64. MUST equal the signer manifest `key_fingerprint`. |
| `algorithm` | string | ┬¦3.1 v0.5 registry name of the **notary's** signing algorithm (`ed25519` for production, `hmac-sha256` TEST-ONLY). |
| `signed_at_iso` | string | RFC 3339 UTC, millisecond precision, trailing `Z`, per `spec/canonical-serialization.md` Rule 4. |
| `signature_b64` | string | The notary's signature over `attestation_canonical_hash`, base64-standard. |

### 2.3 `notaries/<notary_id>.json`

Each referenced `notary_id` resolves to a notary public-key manifest with the
same shape as a signer manifest (`spec/v0.5-signer-trust.md` ┬¦2.3), reusing
`SignerManifest`:

| Field | Type | Description |
|---|---|---|
| `signer_id` | string | MUST equal `notary_id` (the manifest's identity). |
| `algorithm` | string | ┬¦3.1 registry name. |
| `public_key_b64` | string | base64-standard public key bytes. |
| `key_fingerprint` | string | `sha256(<public_key_bytes>)` hex lowercase, length 64. |
| `notary_uri` | string \| null | Unused for a notary's own manifest; MAY be null. |

The notary manifest `key_fingerprint` MUST be recomputed by the verifier;
mismatch ŌåÆ `E_SIGNER_FINGERPRINT_MISMATCH` (same code as v0.5 ┬¦6).

## 3. Verification Path

`resolve_notary_attestation(signer_manifest, attestation, notary_manifest)`
returns `"verified-by-notary"` iff **all** hold; otherwise it raises
`SignerVerificationError` with the indicated code:

1. `attestation.signer_id == signer_manifest.signer_id`
   ŌåÆ else `E_SIGNER_NOTARY_INVALID`.
2. `attestation.key_fingerprint == signer_manifest.key_fingerprint`
   ŌåÆ else `E_SIGNER_NOTARY_INVALID`. (The notary must vouch for the *same*
   key the package signer used; this closes a key-substitution gap.)
3. `attestation.notary_id == notary_manifest.signer_id`
   ŌåÆ else `E_SIGNER_NOTARY_INVALID`.
4. `attestation.algorithm` is in the ┬¦3.1 registry
   ŌåÆ else `E_SIGNER_ALGORITHM_UNKNOWN`.
5. `attestation.algorithm == notary_manifest.algorithm`
   ŌåÆ else `E_SIGNER_ALGORITHM_MISMATCH` (pre-dispatch confused-deputy guard,
   matching v0.5 ┬¦5).
6. The notary manifest `key_fingerprint` recomputes from its `public_key_b64`
   ŌåÆ else `E_SIGNER_FINGERPRINT_MISMATCH`.
7. `attestation.signature_b64` verifies against the notary public key over
   `attestation_canonical_hash` (┬¦2.1) ŌåÆ else `E_SIGNATURE_INVALID`.

A validator that is handed **no** attestation, or that cannot reach the
notary, MUST fall back to `resolve_notary` (v0.5 ┬¦4) and obtain
`"verified-locally"`. When a caller passes `--require-notary` and the result
is `"verified-locally"`, the existing `E_SIGNER_NOTARY_REQUIRED` (v0.5 ┬¦6)
applies unchanged.

## 4. Error Codes (additive)

| Code | Meaning |
|---|---|
| `E_SIGNER_NOTARY_INVALID` | A notary attestation envelope is structurally present but fails an identity / fingerprint binding check in ┬¦3 (signer_id, key_fingerprint, or notary_id mismatch). |

`E_SIGNER_NOTARY_REQUIRED`, `E_SIGNATURE_INVALID`, `E_SIGNER_ALGORITHM_UNKNOWN`,
`E_SIGNER_ALGORITHM_MISMATCH`, and `E_SIGNER_FINGERPRINT_MISMATCH` are reused
from v0.5 ┬¦6 with their existing meanings.

## 5. Compatibility

Fully additive over v0.5. No mandatory new file; packages with no notary
attestation validate exactly as before. `resolve_notary` (the v0.5 stub) is
unchanged and still returns `"verified-locally"`. The `NotaryAttestation`
`Literal` is unchanged (`"verified-locally"` | `"verified-by-notary"`).
