# Aphelion v0.3 Claim Semantics — R1-R4 Reserved Roadmap Slots

**Version:** 0.3.0-r1r4
**Status:** Normative
**Date:** 2026-05-09

> **Naming note**: "v0.3" here refers to the **Aphelion roadmap reserved slots R1-R4** for claim semantics. The existing `spec/migration-v0.3-to-v0.4.md` covers a different historical v0.3→v0.4 migration. To disambiguate this spec's full identifier is `v0.3-r1r4`.

---

## 1. Scope

This spec adds four optional frontmatter fields and one optional array, all reader-side conflict-resolution machinery, to claim frontmatter. No manifest changes. No wire-format bump.

**Roadmap slot:** R1-R4 (per Aphelion canonical roadmap §RESERVED).

**Provenance:** Chris-pinned 2026-05-09 via 8-of-12-substantive xcouncil round; per-item majorities 5-6/8. ADR-0002 carries the per-item verdict trail and the Phase-4 review carve-out for `confidence`.

**Migration:** additive to wire format `2.0`. New optional frontmatter fields (`valid_from`, `valid_until`, `polarity`, `supersedes`) are added with `format-optional` owner tags in `schema-matrix.md`. v0.4 readers ignoring these fields MUST still validate every previously-conforming v0.4 package unchanged. The four-field R4-trigger list (deliberately excluding `confidence`) is final per Chris confirmation 2026-05-09 PM — see §6.5.

### 1.1 New frontmatter fields

| Slot | Field | Type | Required? | Notes |
|---|---|---|---|---|
| R1 | `confidence` | number | format-optional (already existed in v0.4) | Tightened semantics — see §3 |
| R2 | `valid_from` | string (ISO 8601 UTC `Z`) | format-optional | unbounded if absent |
| R2 | `valid_until` | string (ISO 8601 UTC `Z`) | format-optional | unbounded if absent |
| R3 | `polarity` | enum `{affirm, negate, unknown}` | format-optional | defaults to `affirm` if absent |
| R4 | `supersedes` | array<string> (claim_id) | format-optional | lex-sorted + dedupe; cross-package allowed (lenient, see §6) |

### 1.2 New reader-side derivation (NOT a frontmatter field)

`conflict_class` is computed by `AphelionReadAdapter` from R1+R2+R3+`supersedes`+`created_at` and injected into `query_result.payload`. It MUST NOT be written to frontmatter. Writing it to frontmatter is an `E_CLAIM_RESERVED_FIELD` violation.

`conflict_class ∈ {none, scope_mismatch, supersession, contradiction, ambiguity}`.

## 2. Hard Constraints (decided)

1. **Additive only** — no existing v0.4 field is removed or has its meaning changed (except `confidence` 3dp tightening, which already matched the v0.4 spec normative form).
2. **Reader-side derivation** — `conflict_class` and the R4 detection algorithm live in the reader, not in frontmatter.
3. **Determinism** — every derivation is a pure function of inputs. Same package + same `query_time` → same `conflict_class`.
4. **Solo-dev / offline / YAGNI** — no service, no remote validation, no signer changes (v0.5 signer untouched).
5. **R1-R4 outputs ride inside `query_result.payload`** — no envelope changes.

## 3. R1 — `confidence` (semantic tightening of v0.4 field)

| Property | Rule |
|---|---|
| Type | number `0.0 ≤ x ≤ 1.0` |
| Serialization | exactly **3 decimal digits** (e.g. `0.900`, NOT `0.9`) — already normative in v0.4 `claim-frontmatter.md` |
| Default if absent | reader-side `1.000` (NOT written back) |
| Range error | `PX_E_CLAIM_CONFIDENCE_RANGE` if outside `[0.000, 1.000]` |
| Type error | `PX_E_CLAIM_CONFIDENCE_TYPE` if not numeric |
| Precision error | `PX_E_CLAIM_CONFIDENCE_PRECISION` if not exactly 3dp serialization on the wire |

> **Decision note (2026-05-09):** The v0.3 design memo proposed 2dp precision. Repo verification (`spec/claim-frontmatter.md` line 20) showed v0.4 already locked 3dp. v0.3 inherits 3dp to avoid breaking v0.4. — *Evidence-based override of memo recommendation.*

---

## 4. R2 — `valid_from` / `valid_until`

| Property | Rule |
|---|---|
| Type | string (ISO 8601 UTC, `Z` suffix, exactly 20 chars `YYYY-MM-DDTHH:MM:SSZ`) |
| Quoting | MUST be quoted in YAML to prevent date coercion |
| Default | unbounded both ends (absent = no temporal constraint that side) |
| Order constraint | if both present, `valid_from <= valid_until` (else `PX_E_CLAIM_VALIDTIME_ORDER`) |
| Type error | `PX_E_CLAIM_VALIDTIME_TYPE` if non-string |
| Format error | `PX_E_CLAIM_VALIDTIME_FORMAT` if not the strict 20-char format |

### 4.1 R2 query-time semantics

A claim is `R2-active(query_time)` iff `(valid_from is None or valid_from <= query_time) AND (valid_until is None or query_time <= valid_until)`.

`query_time` default — see §6 D1.4 resolution.

---

## 5. R3 — `polarity`

| Property | Rule |
|---|---|
| Type | enum `{affirm, negate, unknown}`, strictly lowercase ASCII |
| Default if absent | `affirm` |
| Type error | `PX_E_CLAIM_POLARITY_TYPE` if non-string |
| Value error | `PX_E_CLAIM_POLARITY_VALUE` if not in enum |

`negate` means the claim asserts the negation of its body (e.g. claim body `"Chris likes coffee"` with `polarity: negate` reads as `"Chris does NOT like coffee"`). `unknown` means the writer abstained.

## 6. R4 — `supersedes` + Conflict Detection

### 6.1 `supersedes` field

| Property | Rule |
|---|---|
| Type | array<string> (claim_id values, UUID v7 format) |
| Sort | lex-sort + dedupe ON WRITE (writer-side canonicalization) |
| Self-reference | `claim_id ∈ supersedes` → `PX_E_CLAIM_SUPERSEDES_SELF` |
| Duplicates within array | `PX_E_CLAIM_SUPERSEDES_DUPLICATE` |
| Cross-package references | **LENIENT** — see §6.2 (D1.2 resolution) |

### 6.2 D1.2 — Cross-package `supersedes` references (Chris-pinned 2026-05-09)

**Verdict: (a) lenient — reader-side ignore-not-found, log warning + Prometheus counter** (5/8 majority: Sonnet, Codex, Laguna, MiniMax, MiMo; 1 b strict, 1 c forbid)

A `supersedes` entry MAY reference a `claim_id` that is not present in the currently loaded package set. In that case:

- Reader emits `W_CLAIM_SUPERSEDES_DANGLING` warning (NOT an error)
- That `supersedes` entry is dropped from the active R4 graph for this query
- The claim itself remains valid and queryable
- The warning carries `{claim_id_with_supersedes, dangling_target_id, package_id}` for ops triage
- **Production deployments MUST expose this as a Prometheus counter** `aphelion_supersedes_dangling_total{package_id, target_id}` and configure an alert on non-zero rate. Log-only is insufficient — typo'd `claim_id` in `supersedes` would otherwise leave the supersession silently un-applied with no human-visible signal.

Rationale: Aphelion is a file format with multi-package corpora. Strict cross-package validation would require loading all referenced packages (impractical). Forbidding cross-package would block legitimate "package A supersedes package B's claim" use cases.

### 6.3 R4 Detection Algorithm — Normative Steps 0-3

These steps are normative and flow directly from D1.5 + D1.2 verdicts.

```
input: candidate_claims, query_time

step 0 (D1.5 enforcement, see §6.5):
  for each claim with R4-trigger field present (polarity | valid_from | valid_until | supersedes):
    if subject is missing:
      raise PX_E_4144 / E_CLAIM_SUBJECT_REQUIRED_FOR_CONFLICT

step 1: active_set = { c in candidate_claims | c.subject == query.subject AND R2-active(c, query_time) }

step 2 (size predicates):
  if active_set is empty: return conflict_class=not_found
  if |active_set| == 1:    return conflict_class=none

step 3 (explicit supersession, lenient cross-pkg per D1.2):
  for c in active_set:
    if c.supersedes is non-empty:
      drop all c' in active_set where c'.claim_id in c.supersedes
      (cross-package targets that aren't loaded → emit W_CLAIM_SUPERSEDES_DANGLING, no drop)
  recurse on shrunk active_set
  if |active_set| == 1 after recursion: return conflict_class=supersession
```

After step 3, if `|active_set| > 1`, the spec leaves the resolution policy to the implementation (subject to the contract in §6.3a below). Non-normative reference logic for resolving the residual set is in §6.3b — those steps are **not** Chris-pinned and are expected to be tuned as test data accumulates.

### 6.3a Residual-set contract

Implementations resolving `|active_set| > 1` after step 3 MUST:

- Return one of `{contradiction, ambiguity, supersession}` (no new conflict_class enums)
- Be deterministic — same `(active_set, query_time)` → same output
- Treat polarity divergence as `contradiction` (NOT `supersession`) — this preserves the D1.6 verdict that "polarity differ" is not sufficient grounds for supersession

The exact tiebreak policy (recency vs confidence vs hybrid) is implementation-defined and SHOULD be documented per-deployment.

### 6.3b Reference resolution (NON-NORMATIVE — implementation sketch)

The autopilot draft proposed the following ordered steps after step 3. They are recorded here for reference but are NOT Chris-pinned. Implementations MAY adopt them, modify them, or substitute their own policy that satisfies §6.3a.

```
ref-step 4 (polarity divergence):
  if polarities in active_set are mixed:
    return conflict_class=contradiction (surface ALL claims)

ref-step 5 (unknown vs definite):
  if active_set has both unknown and {affirm|negate}:
    return conflict_class=ambiguity (surface definite as primary)

ref-step 6 (recency tiebreak):
  sort by created_at desc, claim_id asc → newest wins
  return conflict_class=supersession

ref-step 7 (fallthrough — implementation choice):
  return conflict_class=ambiguity (all surfaced, sorted by confidence desc, claim_id asc)
```

> **Removed from earlier draft:** a "confidence delta > 0.05 → supersession" step preceded the fallthrough. It was unreachable as written (recency tiebreak in ref-step 6 already returned for every case) and the `0.05` threshold was autopilot-original, not from any verdict. Removed; document tiebreak policy if you adopt confidence-based.

> **D1.6 (Chris-pinned 2026-05-09):** **NOT adopted** — Nemotron/Laguna's "polarity differ + containment → supersession" rule is NOT added. Deferred to v0.4 candidate (CHANGELOG). 6/8 majority. Rationale: polarity-based supersession would conflate "contradiction" with "supersession"; v0.3 keeps the two distinct.

### 6.4 D1.4 — `query_time` default (Chris-pinned 2026-05-09)

**Verdict: (b) default = `datetime.now(UTC)`, payload echoes `used_query_time`** (5/8 majority)

```python
# AphelionReadAdapter.query() signature
def query(self, *, subject: str, query_time: datetime | None = None, ...) -> QueryResult:
    raw = query_time if query_time is not None else datetime.now(timezone.utc)
    # Truncate to second precision BEFORE comparison so the echoed value
    # matches the value actually used in R2-active filtering. Any sub-second
    # precision the caller passed is intentionally dropped here, consistently.
    used_query_time = raw.replace(microsecond=0)
    # ... R4 detection runs against used_query_time (NOT raw) ...
    return QueryResult(..., used_query_time=used_query_time.isoformat() + 'Z')
```

The reader MUST truncate `query_time` to whole-second precision **before** R2-active comparison — applied uniformly to both default `now(UTC)` and caller-supplied values. The echoed `used_query_time` in the result payload is the same truncated value, so the contract "log this value, replay later for byte-identical R4 output" holds. Sub-second-precision values are deliberately not honored: R2's `valid_from` / `valid_until` are themselves second-precision (20-char ISO 8601 with no fractional component), so finer granularity would compare against a quantization the spec doesn't carry.

CLI exposes `--query-time now` (default; uses truncated `datetime.now(UTC)`) and `--query-time YYYY-MM-DDTHH:MM:SSZ`.

> Mitigates Codex/MiMo "reproducibility" concern (caller logs `used_query_time`, replays bit-identically) and silent-failure-hunter finding #7 (input/echo divergence under sub-second precision).

### 6.5 D1.5 — `subject` required-when-R4 (Chris-pinned 2026-05-09, primary design call)

**Verdict: (c) `subject` MUST be present if any R4-trigger field is present** (5/8 majority: Sonnet, Qwen, MiMo, Nemotron, GPT-OSS)

**Normative R4-trigger fields** (post-review, see ⚠️ note below): `polarity`, `valid_from`, `valid_until`, `supersedes`.

If a claim has any of those but is missing `subject`, the validator raises:

```
PX_E_4144 / E_CLAIM_SUBJECT_REQUIRED_FOR_CONFLICT
"Claim {claim_id} uses R4 fields ({listed_fields}) but missing required 'subject' for conflict grouping. R4 conflict detection requires a grouping key."
```

Rationale chain:
1. R4 semantic basis is "claims about the same subject may conflict". No subject → no grouping key → conflict detection meaningless.
2. Skip-on-missing (option a) makes R4 fields silently inert when subject is forgotten — confusing for writers.
3. Composite-key fallback (option b) cascades to `(predicate, object)` which are also optional → no terminal condition.
4. Synthetic key = `claim_id` (option d) makes R4 always-no-conflict — kills the feature.

### Backward-compat carve-out — `confidence` excluded (Chris-confirmed 2026-05-09 PM)

The original D1.5 verdict text listed five R4-trigger fields: `confidence | polarity | valid_from | valid_until | supersedes`. Phase-4 review identified that `confidence` is a long-standing v0.4 `format-optional` field (`claim-frontmatter.md` line 20, normative since 2026-04-21). Including it as a trigger would have made every existing v0.4 claim using `confidence` without `subject` a v0.3 validation error — breaking, not additive.

**Resolution (Chris-confirmed 2026-05-09 PM):** `confidence` is excluded from the R4-trigger list. R4-trigger is canonically the four v0.3-net-new fields (`polarity`, `valid_from`, `valid_until`, `supersedes`).

Rationale: `confidence` is metadata about a single claim's certainty; it is a tiebreak input to R4 conflict resolution, NOT an opt-in signal that the claim participates in a conflict graph. The other four fields actively declare "I have a polarity stance" / "I have a temporal scope" / "I supersede something" — those are genuine opt-in signals.

A claim that wants subject-required semantics around `confidence` can always opt in explicitly by also setting `polarity: affirm` or another R4-trigger.

Within the four-field trigger list: v0.4 claims without those fields are unaffected. Only claims that opt into R3/R4 semantics by adding those fields incur the requirement. `subject` itself remains `format-optional` at the schema-matrix level — the conditional check is a v0.3 validator concern, not a wire-format change.

## 7. Reserved-field Guard

The string literal `conflict_class` MUST NOT appear as a frontmatter key. Validator raises:

```
PX_E_CLAIM_RESERVED_FIELD
"Field 'conflict_class' is reserved for reader-side derivation; it cannot be set in frontmatter."
```

Same applies to any future `r4_*` derivation field added by the adapter.


### 8.2 Supersession chain

Claim A (older):
```markdown
---
claim_id: "0193e2b1-0001-7000-8000-00000000aaaa"
confidence: 0.700
created_at: "2026-05-08T00:00:00Z"
polarity: "affirm"
subject: "chris"
type: "user_preference"
# ... required fields ...
---
Chris likes tea.
```

Claim B (newer, supersedes A):
```markdown
---
claim_id: "0193e2b1-0001-7000-8000-00000000bbbb"
confidence: 0.900
created_at: "2026-05-09T10:00:00Z"
polarity: "affirm"
subject: "chris"
supersedes:
  - "0193e2b1-0001-7000-8000-00000000aaaa"
type: "user_preference"
# ... required fields ...
---
Chris likes coffee.
```

Reader output for `query(subject="chris", type="user_preference")`:
- Step 1: active_set = {A, B}
- Step 3: B supersedes A → drop A → active_set = {B}
- Step 2 fallthrough: |active_set|=1 → `conflict_class=none`, primary=B
- payload also carries: `superseded_claims=[A.claim_id]`

### 8.3 Contradiction (polarity divergence)

Same subject, opposite polarity, both active and not in any supersession chain → `conflict_class=contradiction`, both surfaced.

## 9. CLI Additions

### 9.1 `aphe canonicalize` (D1.3 resolution)

**D1.3 verdict (Chris-pinned 2026-05-09): (c) raise + provide CLI** (6/8 majority: Codex, Laguna, Nemotron, Qwen, MiMo, GPT-OSS)

The validator raises `PX_E_4143 / E_CLAIM_KEY_ORDER` for non-canonical frontmatter key order (existing v0.4 spec normative). v0.3 ships an official repair tool:

```
aphe canonicalize <path> [--in-place|--out <path>]

Reorders frontmatter keys to ASCII-codepoint ascending, normalizes confidence
to exactly 3dp serialization, lex-sorts and dedupes supersedes arrays. Does
NOT change semantic content; identity-affecting fields (claim_id, content_hash
inputs) are preserved verbatim.

Exit codes:
  0 — canonicalization applied (or input was already canonical); no PX_E raised
  1 — input could not be parsed; CLI surfaces the parse error verbatim
       (typical underlying codes: PX_E_4006 PARSE_ERROR, PX_E_4010 UTF8_INVALID)
  2 — semantic violation discovered during canonicalization;
       most commonly PX_E_4144 / E_CLAIM_SUBJECT_REQUIRED_FOR_CONFLICT
       or any of the v0.3 R1-R4 type/range/format codes (PX_E_4101..4144).
       The CLI prints the offending PX code and a one-line remediation hint.
```

Why CLI over auto-normalize: auto-normalize on read would silently mask writer bugs and create a divergence between on-disk bytes and content_hash inputs. CLI keeps the contract loud (`E_CLAIM_KEY_ORDER` is an error) but cheap to fix (`aphe canonicalize` round-trip).

### 9.2 R4 inspection

```
aphe r4 inspect <package_path> --subject <subject> [--query-time <iso8601z>]
  → emits {conflict_class, primary, surfaced[], superseded[], used_query_time}
```
